## 二、数据流

```
GitHub Trending + Search API
  ↓
trending.json（每日快照，保留7天）
  ↓
评估引擎（规则 + LLM 打分）
  ↓
candidates.json（评分 + 理由 + 复杂度）
  ↓
用户确认（或自动模式：仅 P0 级别自动实现）
  ↓
实现引擎（创建分支 → 写代码 → 测试 → 审查）
  ↓
QA 验证（npm test 通过）
  ↓
合并到 master + 左脑记忆
  ↓
写入 evolved-features.json（已实现特性追踪库）
  ↓
持续感知层：每天对比已实现特性 vs GitHub 最新趋势
  ↓
发现更好方案 → 升级建议 → 回到实现流程
```

## 三、Step 1 — GitHub 扫描器

### 3.1 数据源（4 个渠道）

| 渠道 | URL | 数据格式 | 说明 |
|:-----|:----|:---------|:-----|
| GitHub Trending | `https://github.com/trending?since=daily` | HTML | 每日热门项目 |
| GitHub Search API | `search/repositories?q=claude+code&sort=stars&order=desc` | JSON | 按关键词搜索 |
| GitHub Search API | `search/repositories?q=claude+code+hooks&sort=stars&order=desc` | JSON | hooks 方向 |
| GitHub Search API | `search/repositories?q=claude+code+agent&sort=stars&order=desc` | JSON | agent 方向 |

### 3.2 搜索关键词

```json
[
  "claude code", "claude-code", "claude code extensions",
  "claude code hooks", "claude code agent", "claude code mcp",
  "claude code automation", "claude code self-improvement",
  "claude memory", "claude code tools",
  "anthropic claude code", "claude code custom"
]
```

### 3.3 核心逻辑

```
1. 抓取 GitHub Trending（HTML 解析，提取 repo name/stars/description）
2. 按关键词搜 GitHub Search API（每个关键词取 top 10）
3. 过滤：近 30 天创建 or 近 7 天有更新
4. 去重 + 按相关度评分
5. 保留 top 20 写入 trending.json
```

### 3.4 相关度评分规则

| 信号 | 分数 | 说明 |
|:-----|:-----|:-----|
| 名称含 `claude-code` | +10 | 直接相关 |
| 名称含 `claude` + `extension/hook/tool` | +8 | 高度相关 |
| 名称含 `mcp` + `claude` | +7 | MCP 相关 |
| topics 含 `claude` | +5 | 标签匹配 |
| stars > 100 | +3 | 热门 |
| stars > 30 | +2 | 有一定热度 |
| 近 7 天更新 | +2 | 活跃项目 |
| 语言 = JS/TS/Shell | +2 | 我们的技术栈 |
| description 含 memory/agent/automation | +3 | 方向匹配 |

## 四、Step 2 — 评估引擎

### 4.1 评估维度

| 维度 | 权重 | 评估标准 |
|:-----|:-----|:---------|
| **实用性** | 30% | 这个功能对我们的工作流有什么帮助？ |
| **可行性** | 25% | 以现有架构能实现吗？改动量多大？ |
| **独立性** | 20% | 是否依赖外部服务？零依赖优先 |
| **风险度** | 15% | 引入后会不会破坏现有系统？ |
| **新鲜度** | 10% | 是不是我们还没有的能力？ |

### 4.2 评估流程

```
每个候选 repo：
  1. 读 README（通过 fetch MCP）
  2. 读 package.json（如果有）
  3. 读核心源码（关键文件）
  4. 规则评估 → 输出：
     - relevance: 0-10（实用性）
     - feasibility: 0-10（可行性）
     - independence: 0-10（独立性）
     - risk: 0-10（风险度，越高越危险）
     - freshness: 0-10（新鲜度）
     - summary: 一句话总结
     - suggestion: "adopt" / "adapt" / "skip"
     - estimated_effort: "small" / "medium" / "large"
```

### 4.3 决策规则

```
composite_score = relevance*0.3 + feasibility*0.25 + independence*0.2 + (10-risk)*0.15 + freshness*0.1

composite >= 7.0 → 🟢 自动采纳（P0）
composite 5.0-6.9 → 🟡 建议采纳（需确认）
composite < 5.0 → 🔴 跳过
```

## 五、Step 3 — 实现引擎

### 5.1 实现策略

| suggestion | 策略 | 说明 |
|:-----------|:-----|:-----|
| adopt | 直接集成 | 功能好且兼容，直接合并 |
| adapt | 改造适配 | 核心思路好但需要适配我们的架构 |
| skip | 记录 | 不实现，但记录"发现了什么" |

### 5.2 实现流程

```
对于每个 adopt/adapt 候选：

1. 创建工作分支
   git checkout -b evolution/feature-name

2. 分析原始代码
   - 读核心逻辑
   - 识别可复用的部分
   - 设计适配方案

3. 实现
   - 按 AiCode 的代码风格
   - 复用现有基础设施（左脑/调度器/MCP）
   - 新增必要的文件

4. 测试
   - npm test（确保现有测试不回归）
   - 新增针对新功能的测试

5. 审查
   - 调用 reviewer agent 审查代码

6. 合并
   - git checkout master
   - git merge evolution/feature-name
   - npm test（最终验证）
```

### 5.3 安全约束

| 规则 | 说明 |
|:-----|:-----|
| ⛔ 不引入 npm 依赖 | 除非绝对必要，否则用零依赖方案 |
| ⛔ 不修改 dispatcher.js 核心 | 新功能不能改变调度逻辑 |
| ⛔ 不删除现有文件 | 只能新增或修改 |
| ✅ 必须通过 npm test | 新旧测试全过 |
| ✅ 必须有测试 | 新功能必须带测试 |
| ✅ 必须有文档 | 更新 README 或最佳实践文档 |

## 八、持续感知层（防闭门造车）⭐

> **核心问题**：我们实现了某个功能，但 GitHub 上已经出现更好的方案了，或者我们学的东西已经过时了。
> **解决思路**：不只"学一次"，而是持续跟踪 GitHub 趋势，对比自己的实现和社区最新方案。

### 8.1 已实现特性追踪库

```
data/github/evolved-features.json

每条记录：
{
  "feature": "向量语义搜索",
  "source_repo": "some/awesome-project",
  "implemented_at": "2026-06-23",
  "github_keywords": ["claude memory", "vector search", "embedding"],
  "last_checked": "2026-06-23",
  "status": "current" | "outdated" | "superseded",
  "alternatives_found": []
}
```

### 8.2 三层感知机制

```
Layer 1: 每日扫描（轻量）
  每天 cron 触发时，不只看新项目，还要对比"已实现特性"的关键词
  → 检查是否有新的替代方案出现
  → 如果有 → 标记为 potential_upgrade

Layer 2: 每周对比（深度）
  每周一次，对每个已实现的特性：
  1. 用该特性的关键词重新搜 GitHub
  2. 对比：新项目 stars 增速 vs 我们当初学的项目
  3. 如果新项目明显更热 → 标记为 outdated
  4. 生成"升级建议"

Layer 3: 月度审计（战略）
  每月一次，分析：
  - 哪些特性社区已经不用了？
  - 哪些方向突然火了但我们还没跟上？
  - 我们的实现和社区最佳实践差距多大？
  → 输出"进化战略报告"
```

### 8.3 过时检测规则

| 信号 | 判定 | 动作 |
|:-----|:-----|:-----|
| 已实现的特性，新搜到同类项目 stars 远超我们学的源头 | 升级信号 | 标记 potential_upgrade |
| 源头项目 90 天无更新 | 可能过时 | 标记 check_needed |
| 社区出现新的实现方式，且多个项目采用 | 趋势转变 | 标记 outdated |
| 我们实现的功能，GitHub 上已有官方/更好的方案 | 可替代 | 标记 superseded |

### 8.4 升级流程

```
potential_upgrade 出现时：
  1. 读取新方案的 README + 源码
  2. 与我们当前实现对比（功能/复杂度/性能）
  3. 评估升级收益 vs 风险
  4. 输出升级建议：
     - 🟢 建议升级（新方案明显更好且迁移简单）
     - 🟡 可选升级（各有优劣，看偏好）
     - 🔴 不需要升级（我们的实现已经够好）
  5. 用户确认后执行升级
```

### 8.5 防退化闭环

```
实现新功能后
  → 自动写入 evolved-features.json
  ↓
每日扫描时
  → 自动对比已实现特性的替代方案
  ↓
每周对比时
  → 生成"哪些可能过时"的报告
  ↓
用户会话启动时
  → 如果有过时警报，主动提示："发现 X 个特性可能有更好方案"
  ↓
用户确认升级
  → 回到 Step 3 实现引擎
  ↓
循环
```

### 8.6 关键原则

| 原则 | 说明 |
|:-----|:-----|
| **不闭门造车** | 每次实现前先看 GitHub 最新方案，不盲目造轮子 |
| **不过度跟风** | 只在确实比我们现有实现更好时才升级 |
| **有记忆** | 每个已实现的特性都有追踪记录，不会忘记"从哪学的" |
| **可回滚** | 任何升级都能回退到上一个版本 |
| **渐进式** | 一次只升级一个特性，验证通过后再升级下一个 |

## 九、进化的进化

这个系统本身也会进化：

1. **搜索关键词会调整** — 根据我们关注的方向，自动增减关键词
2. **评估标准会优化** — 根据过去采纳/拒绝的历史，调整评分权重
3. **实现模板会积累** — 成功实现的功能形成模板，下次类似功能更快实现
4. **进化日志会分析** — 统计：什么类型的项目最值得学？什么方向收益最大？
5. **感知层会自适应** — 发现某个关键词搜不到有价值的结果 → 自动替换关键词

## 十一、使用方式

```bash
# 手动跑一次扫描
node scripts/evolution/daily-evolution.js scan

# 查看候选
node scripts/evolution/daily-evolution.js candidates

# 手动实现某个候选

node scripts/evolution/daily-evolution.js watch
```

## 十三、风险和缓解

| 风险 | 概率 | 缓解措施 |
|:-----|:-----|:---------|
| GitHub API 限流 | 中 | 缓存结果，每天只请求一次，用条件请求 |
| 引入破坏性代码 | 低 | 必须通过 npm test + review agent |
| 扫描到低质量项目 | 高 | 评估引擎打分 + 置信度阈值 |
| 自动实现引入 bug | 中 | 每次实现都走分支 → 测试 → 审查 → 合并 |
| 依赖外部服务 | 低 | 安全约束：零依赖优先 |
| 过度升级 | 中 | 每次只升级一个特性 + 用户确认 |
| 闭门造车 | 高 | 三层感知机制 + 每日对比 GitHub 趋势 |
