## 🚀 自主演进模式（v2.0.0 P0-1）

> 用户提的诉求："你设置一个系统开关是否开启全程自主开发"。

**开关状态**：
- 🤖 自主模式 ON：Claude 自主选下一个增量做，关键决策写入快照不询问
- 🙋 正常模式 OFF（默认）：每完成一个功能 → 询问

**用法**：

```bash
# 打开（带原因）
/autonomous 我离开1小时自动跑
# 或
npm run autonomous:on -- "周末出去办事"

# 关闭
/autonomous-stop
# 或
npm run autonomous:off

# 查看状态
npm run autonomous:status
```

**顶部展示**：session-init Step 7 显式显示当前开关状态。

**行为对比**：

| 场景 | OFF | ON |
|:-----|:----|:---|
| 完成 1 个增量后 | 询问"接下来做啥？" | 自主选下一个 |
| 关键决策 | 询问 | 写入快照继续 |
| commit | 询问 | 自动（安全时） |
| 失败 | 询问 | 5 次后自动停 + 汇报 |

**安全边界**：
- ✅ 自主做：智能增量深化、bug 修、文档、commit
- ⚠️ 慎做：修改 AI 工作目录（commit 前先 snapshot）
- ❌ 不做：push、删分支、删文件、改主目录外文件

**实现**：
- 引擎：`scripts/orchestrator/autonomous.js`
- 状态文件：`.claude/skills/left-brain/memory/autonomous-state.json`
- 命令：`.claude/commands/autonomous.md` + `autonomous-stop.md`
- 测试：38/38 通过（`test-autonomous.js`）

**意义**：让用户能放心离开，Claude 自己继续推进智能演进。这是 v2.0 路线"个人开发的智能增强"第一项。

## 〇、为什么做这件事（核心使命）

### 0.1 起点

用户最初使用 claude + cc-switch + 国内模型这套组合，目标是为了**帮人写代码**。后来发现**单个 Claude agent 不够智能**，所以开始增强。

**核心目标（写入规范，任何时候让 AI 知道）**：
> 围绕增强 Claude 能力这一核心，让 Claude Code 在日常开发中越来越智能、越来越主动、越来越像一个有经验的搭档。

### 0.2 我们的定位（v1.9.1）

我们在 Claude Code 外面包了一层增强壳，让同样的 Claude 模型发挥出 5-10 倍效率：

```
┌─────────────────────────────────────────┐
│           我们的增强层（AiCode）           │
│  调度器 / 记忆库 / 工具链 / 工作流 / 自动化   │
├─────────────────────────────────────────┤
│           Claude Code（官方客户端）         │

```

### 0.3 智能差距诊断（来自 2026-06-24 评估）

| 智能维度 | 当前 AiCode | 主流 Agent（GitHub ⭐10000+） | 差距 |

#### 增量 M9：任务复杂度评分（Dispatcher Quantification）✅ 已完成（M9 · v2.5.0 · 2026-06-25）

**已实现**（M9）：`scoreComplexity(task) → { score, band, breakdown }` 0-10 数字评分，让决策有量化依据。

**实现细节**：
- 位置：`scripts/orchestrator/dispatcher.js`（v2.5.0）
- 三档阈值（`RULES.scoring`）：
  - `score ≤ 3` → `no_dispatch`（不值得派）

## 四、Step 2 — 评估引擎

### 4.1 评估维度

| 维度 | 权重 | 评估标准 |
|:-----|:-----|:---------|
| **实用性** | 30% | 这个功能对我们的工作流有什么帮助？ |
| **可行性** | 25% | 以现有架构能实现吗？改动量多大？ |
| **独立性** | 20% | 是否依赖外部服务？零依赖优先 |
| **风险度** | 15% | 引入后会不会破坏现有系统？ |
| **新鲜度** | 10% | 是不是我们还没有的能力？ |

### 4.2 评估流程

```
每个候选 repo：
  1. 读 README（通过 fetch MCP）
  2. 读 package.json（如果有）
  3. 读核心源码（关键文件）
  4. 规则评估 → 输出：
     - relevance: 0-10（实用性）
     - feasibility: 0-10（可行性）
     - independence: 0-10（独立性）
     - risk: 0-10（风险度，越高越危险）
     - freshness: 0-10（新鲜度）
     - summary: 一句话总结
     - suggestion: "adopt" / "adapt" / "skip"
     - estimated_effort: "small" / "medium" / "large"
```

### 4.3 决策规则

```
composite_score = relevance*0.3 + feasibility*0.25 + independence*0.2 + (10-risk)*0.15 + freshness*0.1

composite >= 7.0 → 🟢 自动采纳（P0）
composite 5.0-6.9 → 🟡 建议采纳（需确认）
composite < 5.0 → 🔴 跳过
```

   git checkout -b evolution/feature-name

   - 读核心逻辑
   - 识别可复用的部分
   - 设计适配方案

   - git checkout master

## 八、持续感知层（防闭门造车）⭐

> **核心问题**：我们实现了某个功能，但 GitHub 上已经出现更好的方案了，或者我们学的东西已经过时了。
> **解决思路**：不只"学一次"，而是持续跟踪 GitHub 趋势，对比自己的实现和社区最新方案。

### 8.1 已实现特性追踪库

```
data/github/evolved-features.json

每条记录：
{
  "feature": "向量语义搜索",
  "source_repo": "some/awesome-project",
  "implemented_at": "2026-06-23",
  "github_keywords": ["claude memory", "vector search", "embedding"],
  "last_checked": "2026-06-23",
  "status": "current" | "outdated" | "superseded",
  "alternatives_found": []
}
```

### 8.2 三层感知机制

```
Layer 1: 每日扫描（轻量）
  每天 cron 触发时，不只看新项目，还要对比"已实现特性"的关键词
  → 检查是否有新的替代方案出现
  → 如果有 → 标记为 potential_upgrade

Layer 2: 每周对比（深度）
  每周一次，对每个已实现的特性：
  1. 用该特性的关键词重新搜 GitHub
  2. 对比：新项目 stars 增速 vs 我们当初学的项目
  3. 如果新项目明显更热 → 标记为 outdated
  4. 生成"升级建议"

Layer 3: 月度审计（战略）
  每月一次，分析：
  - 哪些特性社区已经不用了？
  - 哪些方向突然火了但我们还没跟上？
  - 我们的实现和社区最佳实践差距多大？
  → 输出"进化战略报告"
```

### 8.3 过时检测规则

| 信号 | 判定 | 动作 |
|:-----|:-----|:-----|
| 已实现的特性，新搜到同类项目 stars 远超我们学的源头 | 升级信号 | 标记 potential_upgrade |
| 源头项目 90 天无更新 | 可能过时 | 标记 check_needed |
| 社区出现新的实现方式，且多个项目采用 | 趋势转变 | 标记 outdated |
| 我们实现的功能，GitHub 上已有官方/更好的方案 | 可替代 | 标记 superseded |

### 8.4 升级流程

```
potential_upgrade 出现时：
  1. 读取新方案的 README + 源码
  2. 与我们当前实现对比（功能/复杂度/性能）
  3. 评估升级收益 vs 风险
  4. 输出升级建议：
     - 🟢 建议升级（新方案明显更好且迁移简单）
     - 🟡 可选升级（各有优劣，看偏好）
     - 🔴 不需要升级（我们的实现已经够好）
  5. 用户确认后执行升级
```

### 8.5 防退化闭环

```
实现新功能后
  → 自动写入 evolved-features.json
  ↓
每日扫描时
  → 自动对比已实现特性的替代方案
  ↓
每周对比时
  → 生成"哪些可能过时"的报告
  ↓
用户会话启动时
  → 如果有过时警报，主动提示："发现 X 个特性可能有更好方案"
  ↓
用户确认升级
  → 回到 Step 3 实现引擎
  ↓
循环
```

### 8.6 关键原则

| 原则 | 说明 |
|:-----|:-----|
| **不闭门造车** | 每次实现前先看 GitHub 最新方案，不盲目造轮子 |
| **不过度跟风** | 只在确实比我们现有实现更好时才升级 |
| **有记忆** | 每个已实现的特性都有追踪记录，不会忘记"从哪学的" |
| **可回滚** | 任何升级都能回退到上一个版本 |
| **渐进式** | 一次只升级一个特性，验证通过后再升级下一个 |
