## 🚀 自主演进模式（v2.2.0 P0-1）

> 用户提的诉求："你设置一个系统开关是否开启全程自主开发"。

**开关状态**：
- 🤖 自主模式 ON：Claude 自主选下一个增量做，关键决策写入快照不询问
- 🙋 正常模式 OFF（默认）：每完成一个功能 → 询问

**两种模式**：

| 模式 | 含义 | 适用场景 |
|:-----|:-----|:---------|
| **single** | 完成**当前一个阶段**后自动停止，关闭开关 | "帮我把这阶段做完就停" |
| **always** | 完成一个阶段后**自动开启新阶段**，循环执行 | "我离开几小时，持续跑" |

**用法**（推荐交互菜单）：

```bash
# 交互式方向键菜单（推荐）：↑↓ 选择，↵ 确认，single/always 自动启动 runner
npm run autonomous

# slash 命令选择框：无参数时弹出 single/always/on/off 选择框
/autonomous

# 传统命令行入口
/autonomous single
npm run autonomous:single -- "我离开1小时，先把这阶段做完"

/autonomous always
npm run autonomous:always -- "周末出去办事，自动跑"

npm run autonomous:on -- "周末出去办事"
npm run autonomous:runner
/autonomous-stop
npm run autonomous:off
npm run autonomous:status
```

**顶部展示**：session-init 顶部显式显示当前开关状态与 mode。

**行为对比**：

| 场景 | OFF | single | always |
|:-----|:----|:-------|:-------|
| 完成 1 个增量后 | 询问"接下来做啥？" | 自动停止 | 自动选下一个 |
| 阶段完成后 | 等待用户 | 停止 | 启动新 `claude -p` 继续 |
| 关键决策 | 询问 | 写入快照继续 | 写入快照继续 |
| commit | 询问 | 自动（安全时） | 自动（安全时） |
| 失败 | 询问 | 5 次后自动停 + 汇报 | 5 次后自动停 + 汇报 |

**安全边界**：
- ✅ 自主做：智能增量深化、bug 修、文档、commit
- ⚠️ 慎做：修改 AI 工作目录（commit 前先 snapshot）
- ❌ 不做：push、删分支、删文件、改主目录外文件

**状态同步规则**（v3.0.0+）：**测试、调用、状态、commit** 都同步
- **闸门 0**（测试先行）：先建测试用例，后改实现
- **闸门 1**（调用同步）：`test-self-reflect.js` 调整 → `scripts/reflection/self-reflect.js` 同步
- **闸门 2**（状态同步）：`session-init.sh` Step 5 顶部展示最近 5 条反思
- **闸门 3**（commit 同步）：状态描述与代码实际一致后 commit
- **闸门 4**（user 上下文）：内部用 `git diff` 看变更，确认后才改外

**选题切换规则**（详见 `.claude/rules/autonomous.md`）：
1. 每个阶段/选题完成后 **必须保存快照**（不受 30 分钟间隔限制）
2. 快照中必须包含 **下一个阶段目标**
3. runner 退出当前 \`claude -p\` 子会话，启动 **新的 \`claude -p\` 子会话** 加载快照并继续下一阶段
4. 目的：控制 token 消耗 + 防止上下文污染 + 保持每个选题起点清晰
   - \`/clear\` 无法被脚本调用，因此用“结束旧子会话、启动新子会话”代替手动清理

**实现**：
- 开关引擎：`scripts/orchestrator/autonomous.js`（v2.2.0）
- 循环执行器：`scripts/orchestrator/autonomous-runner.js`（v2.2.0）
- 交互式菜单：`scripts/orchestrator/autonomous-menu.js`（v3.0.6 · 零依赖 · ↑↓ 方向键选择）
- 状态文件：`.claude/skills/left-brain/memory/autonomous-state.json`（新增 `mode` 字段）
- 快照扩展：`.claude/skills/left-brain/scripts/state-snapshot.js`（新增 `stage` 字段）
- 新会话恢复：`.claude/hooks/SessionStart`
- 命令：`.claude/commands/autonomous.md` + `autonomous-stop.md`
- npm scripts：`autonomous` / `autonomous:single` / `autonomous:always` / `autonomous:start` / `autonomous:runner`
- 测试：64/64 通过（`test-autonomous.js`）+ 12/12 通过（`test-autonomous-runner.js`）+ 6/6 通过（`test-autonomous-menu.js`）

**用法**：

```bash
# single 模式：设置开关并执行一个阶段（完成后自动停止）
npm run autonomous:single -- "我离开1小时，先把这阶段做完"

# always 模式：设置开关并循环执行阶段
npm run autonomous:always -- "周末出去办事，自动跑"

# 在已开启状态下启动 runner
npm run autonomous:runner

# 关闭
/autonomous-stop
# 或
npm run autonomous:off

# 查看状态
npm run autonomous:status
```

**意义**：让用户能放心离开，Claude 自己继续推进智能演进。这是 v2.0 路线"个人开发的智能增强"第一项。

### 0.1 起点

用户最初使用 claude + cc-switch + 国内模型这套组合，目标是为了**帮人写代码**。后来发现**单个 Claude agent 不够智能**，所以开始增强。

### 0.2 我们的定位（v1.9.1）

### 0.3 智能差距诊断（来自 2026-06-24 评估）

### 0.7 演进计划的功能怎么来的（v3.0.1 M16 后新增）

> **常见疑问**："演进计划的任务从哪儿来？""什么时候加新任务？""凭啥选这个不选那个？"
> 本节系统化说明。

#### 🎯 演进计划的单一权威源

**所有"当前在做什么 / 下一个候选 / 完成历史"的状态只在 `evolution-plan.json`（gitignore）**。
04.md §0.4 / CHANGELOG / CLAUDE.md / 12 章里程碑表 = **下游展示**。

| 状态 | 文件位置 | 谁写 |
|:-----|:---------|:-----|
| **事实源** | `.claude/skills/left-brain/memory/evolution-plan.json` | `evolution-lock.js` + `queue-bridge.js` |
| 展示源 | `04.md §0.4 + §十二`、`CHANGELOG.md`、`CLAUDE.md` | AI 同步 |
| 候选源 | `data/github/candidates.json`、`04.md backlog 段`、`.claude/audits/audit-*.md` | `/evolve`、`/audit`、手动 |

#### 📥 4 个候选来源（M16 后完整闭环）

```
候选来源（4 个）                  半自动汇聚              单一权威源
───────────────────              ──────────              ──────────
1. 04.md §0.4 增量定义        ──┐
                                  │
2. /evolve candidates.json     ──┼──→ queue-bridge.js ──→ evolution-plan.json
   (suggestion=adopt)            │    (npm run queue:sync)  (next 队列)
3. /audit backlog 段          ──┤
                                  │
4. 左脑偏好/纠正记忆          ──┘
```

| 来源 | 触发命令 | 产物 | 谁负责 |
|:-----|:---------|:-----|:-------|
| **04.md §0.4 增量定义** | 用户设计（AI 起草 + 用户确认）| 增量段（含验收 / ROI / 关联）| 用户主导 |
| **/evolve GitHub 扫描** | `npm run evolve:scan` + `:analyze` | `data/github/candidates.json` | AI 跑 + 用户审 |
| **/audit 工程自审** | `npm run audit` | 04.md 末尾 backlog 段 + `.claude/audits/audit-*.md` | AI 跑 + 用户整合 |
| **左脑偏好/纠正** | 用户对话中纠正 AI | `left-brain.sh preference "..."` | 用户主导 |

**关键洞察**：前 3 个是"被动产候选"，第 4 个是"主动产候选"。**bridge 半自动汇聚 + 用户最终审核 = 单一权威源不漂移**。

#### 🔍 怎么判断"该不该加进计划"

加进 `next` 队列前，bridge 自动 + 用户手动判断：

| 维度 | 自动判断 | 手动判断 |
|:-----|:---------|:---------|
| **相关性** | /evolve 用 TF-IDF 打分（composite ≥ 7.0 标 `adopt`）| 用户看描述是否符合自身需求 |
| **可落地性** | /evolve 评 `feasibility`（人手/技术栈匹配）| 用户看是否在自己能力圈内 |
| **优先级** | /audit 分 P0/P1/P2（按 ROI）| 用户看当下忙不忙、是否阻塞其他任务 |
| **重复性** | bridge dedupe（按 id 严格匹配）| 用户看是否已在 history |
| **是否解决真实问题** | ❌ 不判断 | ✅ **必须用户判断**（机器不知道你的痛点）|

#### 🚦 加进队列的触发条件（任一满足）

1. **04.md §0.4 设计时**：用户明确指定（如"M17 接下来要做 token 预算"）
2. **/evolve 跑出 `adopt`**：bridge 入队（EVOLVE- 前缀）
3. **/audit backlog P0/P1 段**：用户确认整合后 bridge 入队（AUDIT- 前缀）
4. **用户对话中提到**：直接 `evolution-lock.js queue <id> <title>` 或让我帮你入队

#### ❌ 不加进队列的情况

- /evolve 跑出 `adapt`（要改造才能用）— 等改造方案明确再加
- /evolve 跑出 `skip`（分数低）— 不加
- /audit backlog P2（远期）— 暂存 04.md backlog 段，不入队
- candidates.json 里 status='adopted' 但 30 天未实现 — **bridge 不自动清**，但用户可在下次 audit 时降级

#### 🛠 怎么"加一个新任务"（用户视角）

```bash
# 方法 1：04.md §0.4 加增量段（最规范）
# 写好 "## 增量 M17：..." + 验收 + ROI，然后告诉我 "M17 入队"
# 我会调 evolution-lock.js queue M17 "..." 把 ID 加进 next

# 方法 2：直接命令（最快）
node scripts/orchestrator/evolution-lock.js queue M17-token-budget "token 预算管理" -p P1

# 方法 3：让 bridge 跑一次（汇聚已有候选）
npm run queue:sync:dry   # 先看会入队哪些
npm run queue:sync        # 确认入队
```

#### 🔁 怎么"完成一个任务"

```bash
# 方法 1：让我做（M17 → M18 完成后）
# 我会自动调 evolution-lock.js complete M17 "交付摘要"
# 同时写 history 段 + 同步 04.md / CHANGELOG / 12 章里程碑表

# 方法 2：手动完成
node scripts/orchestrator/evolution-lock.js complete M17 "实现摘要..."
```

#### 📊 队列状态查询

```bash
node scripts/orchestrator/evolution-lock.js status   # 当前锁
node scripts/orchestrator/evolution-lock.js peek M17  # 看 M17 详情
node scripts/orchestrator/evolution-lock.js queue <id> <title>  # 入队
node scripts/orchestrator/evolution-lock.js complete <id> "summary"  # 完成
npm run queue:sync:dry    # 汇聚候选（不入队）
npm run queue:sync        # 汇聚候选（实际入队）
```

## 四、Step 2 — 评估引擎

### 4.1 评估维度

| 维度 | 权重 | 评估标准 |
|:-----|:-----|:---------|
| **实用性** | 30% | 这个功能对我们的工作流有什么帮助？ |
| **可行性** | 25% | 以现有架构能实现吗？改动量多大？ |
| **独立性** | 20% | 是否依赖外部服务？零依赖优先 |
| **风险度** | 15% | 引入后会不会破坏现有系统？ |
| **新鲜度** | 10% | 是不是我们还没有的能力？ |

### 4.2 评估流程

```
每个候选 repo：
  1. 读 README（通过 fetch MCP）
  2. 读 package.json（如果有）
  3. 读核心源码（关键文件）
  4. 规则评估 → 输出：
     - relevance: 0-10（实用性）
     - feasibility: 0-10（可行性）
     - independence: 0-10（独立性）
     - risk: 0-10（风险度，越高越危险）
     - freshness: 0-10（新鲜度）
     - summary: 一句话总结
     - suggestion: "adopt" / "adapt" / "skip"
     - estimated_effort: "small" / "medium" / "large"
```

### 4.3 决策规则

```
composite_score = relevance*0.3 + feasibility*0.25 + independence*0.2 + (10-risk)*0.15 + freshness*0.1

composite >= 7.0 → 🟢 自动采纳（P0）
composite 5.0-6.9 → 🟡 建议采纳（需确认）
composite < 5.0 → 🔴 跳过
```

## 五、Step 3 — 实现引擎

### 5.1 实现策略

| suggestion | 策略 | 说明 |
|:-----------|:-----|:-----|
| adopt | 直接集成 | 功能好且兼容，直接合并 |
| adapt | 改造适配 | 核心思路好但需要适配我们的架构 |
| skip | 记录 | 不实现，但记录"发现了什么" |

### 5.2 实现流程

```
对于每个 adopt/adapt 候选：

1. 创建工作分支
   git checkout -b evolution/feature-name

2. 分析原始代码
   - 读核心逻辑
   - 识别可复用的部分
   - 设计适配方案

3. 实现
   - 按 AiCode 的代码风格
   - 复用现有基础设施（左脑/调度器/MCP）
   - 新增必要的文件

4. 测试
   - npm test（确保现有测试不回归）
   - 新增针对新功能的测试

5. 审查
   - 调用 reviewer agent 审查代码

6. 合并
   - git checkout master
   - git merge evolution/feature-name
   - npm test（最终验证）
```

### 5.3 安全约束

| 规则 | 说明 |
|:-----|:-----|
| ⛔ 不引入 npm 依赖 | 除非绝对必要，否则用零依赖方案 |
| ⛔ 不修改 dispatcher.js 核心 | 新功能不能改变调度逻辑 |
| ⛔ 不删除现有文件 | 只能新增或修改 |
| ✅ 必须通过 npm test | 新旧测试全过 |
| ✅ 必须有测试 | 新功能必须带测试 |
| ✅ 必须有文档 | 更新 README 或最佳实践文档 |

## 八、持续感知层（防闭门造车）⭐

> **核心问题**：我们实现了某个功能，但 GitHub 上已经出现更好的方案了，或者我们学的东西已经过时了。
> **解决思路**：不只"学一次"，而是持续跟踪 GitHub 趋势，对比自己的实现和社区最新方案。

### 8.1 已实现特性追踪库

```
data/github/evolved-features.json

每条记录：
{
  "feature": "向量语义搜索",
  "source_repo": "some/awesome-project",
  "implemented_at": "2026-06-23",
  "github_keywords": ["claude memory", "vector search", "embedding"],
  "last_checked": "2026-06-23",
  "status": "current" | "outdated" | "superseded",
  "alternatives_found": []
}
```

### 8.2 三层感知机制

```
Layer 1: 每日扫描（轻量）
  每天 cron 触发时，不只看新项目，还要对比"已实现特性"的关键词
  → 检查是否有新的替代方案出现
  → 如果有 → 标记为 potential_upgrade

Layer 2: 每周对比（深度）
  每周一次，对每个已实现的特性：
  1. 用该特性的关键词重新搜 GitHub
  2. 对比：新项目 stars 增速 vs 我们当初学的项目
  3. 如果新项目明显更热 → 标记为 outdated
  4. 生成"升级建议"

Layer 3: 月度审计（战略）
  每月一次，分析：
  - 哪些特性社区已经不用了？
  - 哪些方向突然火了但我们还没跟上？
  - 我们的实现和社区最佳实践差距多大？
  → 输出"进化战略报告"
```

### 8.3 过时检测规则

| 信号 | 判定 | 动作 |
|:-----|:-----|:-----|
| 已实现的特性，新搜到同类项目 stars 远超我们学的源头 | 升级信号 | 标记 potential_upgrade |
| 源头项目 90 天无更新 | 可能过时 | 标记 check_needed |
| 社区出现新的实现方式，且多个项目采用 | 趋势转变 | 标记 outdated |
| 我们实现的功能，GitHub 上已有官方/更好的方案 | 可替代 | 标记 superseded |

### 8.4 升级流程

```
potential_upgrade 出现时：
  1. 读取新方案的 README + 源码
  2. 与我们当前实现对比（功能/复杂度/性能）
  3. 评估升级收益 vs 风险
  4. 输出升级建议：
     - 🟢 建议升级（新方案明显更好且迁移简单）
     - 🟡 可选升级（各有优劣，看偏好）
     - 🔴 不需要升级（我们的实现已经够好）
  5. 用户确认后执行升级
```

### 8.5 防退化闭环

```
实现新功能后
  → 自动写入 evolved-features.json
  ↓
每日扫描时
  → 自动对比已实现特性的替代方案
  ↓
每周对比时
  → 生成"哪些可能过时"的报告
  ↓
用户会话启动时
  → 如果有过时警报，主动提示："发现 X 个特性可能有更好方案"
  ↓
用户确认升级
  → 回到 Step 3 实现引擎
  ↓
循环
```

### 8.6 关键原则

| 原则 | 说明 |
|:-----|:-----|
| **不闭门造车** | 每次实现前先看 GitHub 最新方案，不盲目造轮子 |
| **不过度跟风** | 只在确实比我们现有实现更好时才升级 |
| **有记忆** | 每个已实现的特性都有追踪记录，不会忘记"从哪学的" |
| **可回滚** | 任何升级都能回退到上一个版本 |
| **渐进式** | 一次只升级一个特性，验证通过后再升级下一个 |

## 九、进化的进化

这个系统本身也会进化：
