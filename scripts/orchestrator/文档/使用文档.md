# 智能调度模块使用文档

> 让 Claude Code 在每次提问时**自动判断**是否需要派子代理并行执行，实现 **2-3 倍提速**。

---

## 🎯 它是什么

一个轻量级智能调度层，集成在 Claude Code 工作流中：

```
你输入任务
    ↓
PreToolUse 钩子触发 dispatcher.js
    ↓
┌─────────────────────────────┐
│ Layer 1: 规则引擎（12/12）  │  ← 关键词 + 文件数 + 任务类型
│ Layer 2: LLM 评分（预留）   │  ← 灰区任务用小模型打分
└─────────────────────────────┘
    ↓
决策：派 N 个 Agent（1-3）
    ↓
并行执行 → 主会话汇总
```

**核心价值**：

| 模式 | 耗时 | 提升 |
|:-----|:-----|:-----|
| 串行（主会话自己查） | 3-4 分钟 | 基准 |
| 并行（派 2-3 Agent） | 1-2 分钟 | **2-3 倍** |

**实测案例**：排查"点餐系统添加菜品 BUG" → 派 2 个 Agent 找到 6 个真实问题（前端 placeholder、catch 吞错误、后端无 category_id 校验等）。

## 🚀 三种使用方式

### 方式 1：自动（推荐）— PreToolUse 钩子

**已配置**到 `.claude/settings.local.json`，**每次你提问自动跑**：

```json
{
  "matcher": "UserPromptSubmit",
  "hooks": [{
    "type": "command",
    "command": "node H:/AI-han/AiCode/scripts/orchestrator/hooks/dispatch-decision.js",
    "timeout": 3,
    "async": true
  }]
}
```

**效果**：CC 日志里会出现 `dispatch-decision` 输出（如 `dispatch: true, agents: 2`），主会话参考决策派 Agent。

**优点**：零感知、零成本决策。
**限制**：钩子只能"建议"，不能强制派发（需要主会话执行）。

### 方式 2：手动 — /dispatch 命令

输入：
```
/dispatch 排查订单添加菜品失败 BUG
```

**自动执行**：
1. 跑 dispatcher.js 拿决策
2. 派 N 个 Agent 并行
3. 主会话汇总结果

**优点**：可控、可复现。
**适用**：明确知道要派 Agent 的任务。

### 方式 3：纯 CLI — 直接跑规则

```bash
node H:/AI-han/AiCode/scripts/orchestrator/dispatcher.js "你的任务"
```

**输出**：
```json
{
  "dispatch": true,
  "agents": 2,
  "reason": "预估涉及 2 个模块",
  "layer": 1,
  "confidence": 0.9
}
```

**适用**：调试规则、验证决策。

## 📊 规则引擎决策表

### 派 Agent（`dispatch=true`）

| 触发条件 | 优先级 |
|:---------|:-------|
| 命中关键词：`全面` / `彻底` / `完整` / `并行` / `同时` / `一起` / `多模块` / `跨模块` / `全栈` / `前后端一起` | 高 |
| 预估涉及文件 ≥ 5 | 高 |
| 预估涉及模块 ≥ 2 | 高 |
| 任务类型：`bug_fix` / `refactor` / `feature_full` / `migration` / `multi_module` | 中 |

### 不派 Agent（`dispatch=false`）

| 触发条件 | 优先级 |
|:---------|:-------|
| 命中关键词：`快速` / `简单` / `小` / `只改` / `一个一个` / `先 X 再 Y` / `帮我看下` / `解释` / `是什么` / `为什么` / `怎么用` / `如何` / `对比` / `区别` / `推荐` / `建议` | **最高**（优先于任务类型） |
| 预估涉及文件 ≤ 2 | 中 |
| 任务类型：`explanation` / `question` / `single_edit` | 中 |

### 灰区（`dispatch=null`）

以上都不命中 → 默认保守派 1-2 个 Agent（成本敏感度低，先看效果）。

## 🤖 派 Agent 的标准模板

### 模板 A：2 个 Agent（最常见）

```python
# Agent 1: 代码/技术层
Agent(subagent_type="Explore", prompt=f"""
作为代码分析专家，排查任务的【代码层面】：

任务: {TASK}
要求:
- 重点查看相关源代码文件
- 用 Glob/Grep 定位关键路径
- 输出可疑代码片段（带行号）
- 给出修复建议

⚠️ 严格约束:
- 只能 Read/Glob/Grep
- 不能 Edit/Write/Bash
- 输出中文
""")

# Agent 2: 数据/配置/环境层
Agent(subagent_type="Explore", prompt=f"""
作为数据/配置分析专家，排查任务的【数据层面】：

任务: {TASK}
要求:
- 检查 .env / 配置文件
- 检查数据库表结构 / 缓存配置
- 检查 API 接口规范
- 输出问题清单（带证据）

⚠️ 严格约束同上
""")
```

### 模板 B：3 个 Agent（大型任务）

```python
# Agent 1: 代码层
Agent(...prompt="源码分析...")

# Agent 2: 数据层
Agent(...prompt="数据库/缓存分析...")

# Agent 3: 环境层
Agent(...prompt="配置/部署/依赖分析...")
```

### 模板 C：1 个 Agent（轻量并行）

```python
Agent(subagent_type="Explore", prompt=f"分析 {TASK} 的整体结构，输出关键发现")
```

## 🛠️ 调优指南

### 调整规则（不需重启）

编辑 `scripts/orchestrator/dispatcher.js` 中的 `RULES` 对象：

```javascript
const RULES = {
  should_dispatch: {
    keywords: ['全面', '彻底', ...],   // ← 加新关键词
    file_estimate_min: 5,             // ← 调整文件阈值
    max_agents: 3,                    // ← 调整最大 Agent 数
  },
  dont_dispatch: {
    keywords: ['快速', '简单', ...],  // ← 加新关键词
    file_estimate_max: 2,             // ← 调整文件阈值
  },
};
```

**改完立即生效**（钩子会读最新代码）。

### 跑测试验证

```bash
cd H:/AI-han/AiCode/scripts/orchestrator
node test-dispatcher.js
```

应输出 `12/12 通过`。

### 添加新测试用例

编辑 `test-dispatcher.js` 中的 `TESTS` 数组。

### 调整 Agent 数量上限

修改 `dispatcher.js` 中 `max_agents: 3` → 你的新上限。

## 📈 监控与成本

### 当前设定

| 参数 | 值 | 说明 |
|:-----|:---|:-----|
| 最大并行 Agent | 3 | 用户设定 |
| 每次并行 Token 上限 | 不限 | 用户设定（先看效果） |
| 钩子超时 | 3 秒 | 不阻塞主会话 |

### 监控方法

跑 N 次后看 `~/.claude/logs/` 里的钩子输出：
- `dispatch: true` 频率
- 平均 `agents` 数量
- 实际 Token 用量

### 调整建议

| 现象 | 调整 |
|:-----|:-----|
| 派太频繁 | 加 `dont_dispatch` 关键词 |
| 派太少 | 加 `should_dispatch` 关键词 |
| 灰区太多 | 调 LLM 评分阈值（Layer 2） |
| 误判多 | 细化文件/模块数估算 |

## ❓ FAQ

### Q1：为什么不直接每次都派？

**A**：成本。每个 Agent 是独立 context，Token 翻倍。小任务派出去反而更慢（启动开销 > 实际收益）。

### Q2：为什么是 Explore 类型？

**A**：只读。子代理不能修改文件，避免误操作。汇总后再由主会话决定改什么。

### Q3：能强制派多少个吗？

**A**：可以。在提示里说"**并行**"或"**全面**"会触发强信号，直接派 2-3 个。

### Q4：钩子会不会卡主会话？

**A**：不会。`async: true` + `timeout: 3` 确保主流程不被阻塞。失败兜底返回"不派"。

### Q5：能不能让多模型分工？

**A**：当前**不能**。Claude Code 单会话只能用一种模型。要多模型需要换 OpenCode + OMO（路线不同）。

### Q6：Layer 2 LLM 评分什么时候上？

**A**：当前 Layer 1 准确率已经够用（12/12）。如果灰区误判多再考虑。先收集实际数据。

### Q7：能加单元测试吗？

**A**：`test-dispatcher.js` 已经是 12 个测试用例。CI 集成时改成 Jest 即可。

### Q8：规则引擎能学吗？

**A**：当前是纯规则。**理论上**可以加反馈机制（决策错 → 调规则），但需要样本积累。
