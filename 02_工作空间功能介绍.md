## 🧠 二、核心能力详解

### 0. 核心定位：客户端 Agent 增强，不是大模型增强

**我们不做的事**：
- 训练/微调 Claude 模型
- 修改 Anthropic 的模型权重
- 替代 Claude 的推理能力

**我们做的事**：
- 在 Claude Code 客户端外面包了一层"增强壳"
- 让 Claude Code 这个 Agent 更会"使唤" Claude 模型

```
┌─────────────────────────────────────────┐
│           我们的增强层（AiCode）           │
│  调度器 / 记忆库 / 工具链 / 工作流 / 自动化   │
├─────────────────────────────────────────┤
│           Claude Code（官方客户端）         │
│  Agent 运行时 / Hook 系统 / 工具调用        │
├─────────────────────────────────────────┤
│           Claude 大模型（Anthropic）        │
│  推理 / 编码 / 理解 / 生成                 │
└─────────────────────────────────────────┘
```

#### 0.1 本质区别

| 维度 | 大模型增强 | 客户端 Agent 增强（我们的工作） |
|:-----|:-----------|:------------------------------|
| **改什么** | 模型权重、训练数据、微调、Prompt 工程、RAG | 调度、记忆、工具、工作流、UI、自动化 |
| **在哪里跑** | 云端/GPU | 本地 Claude Code CLI |
| **要不要算力** | 需要大量算力 | 几乎不要 |
| **谁能做** | 需要模型训练能力 | 普通开发者用脚本就能做 |
| **我们的项目** | ❌ 不做 | ✅ 全做 |

#### 0.2 每个系统对应 Agent 的哪个部分

| 我们的系统 | 属于 Agent 的哪个组件 | 作用 |
|:-----------|:----------------------|:-----|
| `dispatcher.js` 智能调度 | **编排层（Orchestration）** | 决定什么任务派子 Agent，什么自己干 |
| `left-brain` 记忆系统 | **长期记忆（Long-term Memory）** | 跨会话保存知识、偏好、决策 |
| `mcp.json` + MCP servers | **工具链（Tools）** | 给 Claude 提供本地文件/SQLite/网页抓取能力 |
| `hooks/` + `settings.local.json` | **环境集成（Environment Integration）** | 会话启动/结束/工具调用前自动执行逻辑 |
| `自我约束规范.md` | **工作流（Workflow）** | 标准化 AI 改完代码后的收尾流程 |
| `自我进化循环系统` | **学习反馈（Learning/Feedback）** | 让 Agent 从 GitHub 社区学习新能力 |
| `.claudeignore` + `/compact` | **上下文管理（Context Management）** | 控制传给模型的上下文，省 token |
| `qa-reviewer` + 测试 | **验证层（Validation/Safety）** | 独立验证代码质量，防止乱改 |
| `/evolve`, `/dispatch` 命令 | **用户体验层（UX）** | 给用户更高效的交互入口 |
| `会话快照` + `00_ROOT_快速加载会话.md` | **状态恢复（State Recovery）** | 让 Agent 能跨会话继续工作 |

#### 0.3 为什么这个定位更务实

**大模型我们做不了**：
- 训练/fine-tune 需要算力和数据
- 我们不可能比 Anthropic 更懂 Claude
- 模型能力由官方决定，我们只能等更新

**客户端 Agent 有巨大优化空间**：

| 官方能力 | 缺什么 | 我们补什么 |
|:---------|:-------|:-----------|
| 能对话 | 不会自己决定要不要派 Agent | 智能调度 |
| 有上下文窗口 | 跨会话失忆 | 左脑记忆 |
| 能调工具 | 没有本地记忆/数据库工具 | MCP sqlite/fetch |
| 能改文件 | 改完不自动测试/归档 | 自我约束 |
| 能联网 | 不会主动学习社区方案 | 自我进化 |
| 能并行 | 多个任务容易冲突 | worktree 隔离 |

#### 0.4 客户端增强是乘数效应

同样的 Claude 模型：
- 裸用 Claude Code = 1x
- + 智能调度 = 2-3x
- + 记忆 + 工具 + 工作流 = 5x
- + 自我进化 = 持续增长

#### 0.5 对比主流 Agent + LLM

| 维度 | 主流 Agent + LLM | 我们的 AiCode |
|:-----|:-----------------|:--------------|
| 学习能力 | 固定训练数据 | ✅ 每日扫描 GitHub 自我进化 |
| 调度策略 | 无/简单 | ✅ 规则引擎智能决策 |
| 跨会话记忆 | 对话历史 | ✅ 左脑知识库 + 图谱 |
| 收尾质量 | 需人工提醒 | ✅ 自我约束自动收尾 |
| 工程闭环 | 直接改文件 | ✅ 计划→迭代→归档 |
| 并行隔离 | 共享文件系统 | ✅ worktree 真隔离 |
| 成本控制 | 容易 token 爆炸 | ✅ 上下文分片 + compact |
| 稳定性 | 单点依赖 LLM | ✅ 多层失败兜底 |
| 工具链 | 云端/无 | ✅ 本地 MCP server |
| 会话恢复 | 无 | ✅ 快照索引 + 热启动 |
| QA 验证 | 自测 | ✅ 独立 QA 子代理 |

#### 0.6 用打游戏比喻

- **大模型** = 游戏角色的基础属性（力量/智力）
- **客户端 Agent** = 玩家的操作界面、快捷键、宏、背包管理、自动寻路、任务追踪
- **我们的工作** = 给 Claude 写了一套 **顶级玩家插件**

我们不改角色属性，但让这个角色会：
- 自动判断什么时候该组队（调度）
- 记住每个 NPC 的喜好（记忆）
- 自动整理背包（上下文管理）
- 每天看攻略论坛学习新打法（自我进化）
- 打完副本自动存档（快照）
- 任务失败自动读档回滚（失败兜底）

#### 0.7 长期价值

**可迁移性**：今天增强 Claude Code，明天同样逻辑可迁移到 Cursor/Windsurf 等 Agent 客户端。

**模型无关**：Claude 4 出来直接受益，换 GPT-5 核心逻辑也能复用。

**复利效应**：每实现一个新能力，系统本身更强，知识/模板/规则不断积累。

### 2.1 智能调度（v1.2.0）

**核心问题**：复杂任务让 AI 一个个查很慢，能不能**自动派 2-3 个 Agent 并行**？

**解决方案**：抖音视频同款"主从并行加速"

```
你输入"排查 BUG"
       ↓
dispatcher.js 分析（规则引擎 12 项匹配）
       ↓
判断：派 2 个 Agent（后端 + 前端）
       ↓
并行执行（不互相等）
       ↓
主会话等所有完工 → 汇总 → 输出
```

**实测提速**：
- 串行：3-4 分钟
- 并行：1-2 分钟
- **提速 2-3 倍**

**11 项 v1.2 改进**：
- 灰区 `suggested_action` 字段
- 累加权重（多关键词叠加）

- `confidence` 改 0-1 数字

### 2.2 快照系统（v1.1，v1.9 升级：模式可控）

**核心问题**：会话太长 context 爆炸，下次开会话忘记上次的进度；快照保存"频繁但不可控"。

**解决方案**：一键备份 + 一秒恢复 + **4 模式会话级切换**

```
# 结束会话前（一句话）
node scripts/会话快照/save.js "任务标题" "中文标签" -m "下一步"

# 复杂任务：开干前先关，写完一个功能再开
/snap-mode off                    # 关闭自动保存
/snap-mode manual                 # 只手动 save.js
/snap-mode milestone              # 命中完成/交付才存（默认）
/snap-mode auto                   # 每次 Stop hook 都存
/snap-mode reset                  # 回到全局配置

# 下次开会话（一键）
node scripts/会话快照/load.js v1.1    # 按版本匹配
cat 00_ROOT_快速加载会话.md                # 看根目录索引
```

**4 模式详解**：

| 模式 | Stop hook | 手动 save.js | 显式调用 + --force |
|:-----|:----------|:-------------|:--------------------|
| `off` | 不存 | 不存 | **存** |
| `manual` | 不存 | 存 | 存 |
| `milestone` | 仅含"完成/里程碑/交付"等关键词时存 | 仅含关键词时存 | 存 |
| `auto` | 每次都存 | 存 | 存 |

**优先级链**：`.claude/session-snap-mode.json`（会话级） > `.claude/snapshot-config.json`（全局） > 默认 milestone

**自动化**：
- save.js 自动追加 00_ROOT_快速加载会话.md 的"快照列表"表格
- 自动加 ⭐ 最新 标记
- 自动加"快速启动命令"代码块
- 检测"完成/里程碑/交付"标签 → 加三级检查点提示
- **v1.9 修复**：milestone 关键词检查对显式调用也生效（之前显式调用绕过，是个 bug）

**状态栏实时可见**：

### 2.3 三级检查点（v1.3）

**核心问题**：单独文本备份不够，需要"计划→迭代→归档"完整闭环。

**解决方案**：三段式流程

```bash
# 阶段 1：开始任务 → 计划快照
node scripts/parallel/plan-snapshot.js "任务" "描述" "mermaid" "文件1,文件2"

# 阶段 2：worker 完成 → 迭代快照
node scripts/会话快照/save.js "标题" "标签" -m "下一步"

# 阶段 3：全部完成 → 全局归档
bash scripts/parallel/global-archive.sh "任务"
# 生成 archives/<任务>-<时间戳>/ 目录
# 含：snapshots-index.md + COMPLETION-REPORT.md + Git tag
```

**自动提示**：save.js 检测"完成/里程碑/交付"标签 → 启动段代码块自动加"💡 三级检查点提示"

### 2.4 Git Worktree 多工作区（P0）

**核心问题**：多 worker 并行时文件修改会冲突。

**解决方案**：每个 worker 独立 worktree 目录 + Git 分支

```bash
# 创建 2 个并行 worktree（在 .worktrees/ 下）
bash scripts/parallel/worktree-parallel.sh "任务名" 2

# 合并回主分支
bash scripts/parallel/worktree-merge.sh "任务名"
# 自动删除 worktree + 分支
# 创建 Git tag: archive-任务名-时间戳
```

**实测**：
- 串行：11 秒
- 并行：7 秒
- **提速 36%**

### 2.5 自我约束规范（v1.3.1）

**核心问题**：AI 改完代码后，你每次都得提醒"存快照"、"更新文档"。

**解决方案**：AI 完成后**自动**按规范执行收尾

**改动分级**：

| 级别 | 触发 | 自动动作 |
|:-----|:-----|:---------|
| 🟢 微小 | typo/注释 | 跳过 |
| 🟡 小 | bug fix/参数 | 测试 + 快照 + KB |
| 🔴 大 | 新功能/架构 | + 更新文档 |
| 🏁 里程碑 | v1.X 完成 | + 全局归档 |

**触发关键词**：
- 用户说"做完了/OK/完成/搞定" → AI 自动跑流程
- 用户没说但改了 ≥ 1 文件 → AI 主动跑（不询问）

**位置**：
- 完整规范：`scripts/orchestrator/自我约束规范.md`
- 自动加载：`.claude/rules/self-discipline.md`

### 2.6 自我进化系统（v1.8）⭐⭐⭐

**核心问题**：我们怎么让 Claude Code 不闭门造车，持续从 GitHub 社区学习最新的增强能力？

**解决方案**：AI 每天自动扫描 GitHub，学习爆款 Claude 相关项目，分析可行性，然后本地实现。

```
你什么都不用做
       ↓
每天凌晨 2:00 cron 自动触发
       ↓
github-scanner.js 扫描 GitHub Trending + Search API
       ↓
feature-analyzer.js 5 维度评估候选
       ↓
输出 candidates.json（ adopt / adapt / skip ）
       ↓
下次会话启动时提示："发现 N 个可进化特性，是否查看？"
       ↓
用户确认后 implementer.js 创建分支 → 生成实现 prompt → 编码 → 测试 → 合并
       ↓
写入 evolved-features.json 并左脑记忆
       ↓
trend-watcher.js 持续感知，防止过时
       ↓
循环
```

**5 维度评估**：

| 维度 | 权重 | 说明 |
|:-----|:-----|:-----|
| 实用性 | 30% | 对工作流有什么帮助 |
| 可行性 | 25% | 现有架构能否实现 |
| 独立性 | 20% | 是否依赖外部服务 |
| 风险度 | 15% | 会不会破坏现有系统 |
| 新鲜度 | 10% | 是不是我们还没有的能力 |

**决策规则**：
- 综合分 ≥ 7.0 → 🟢 自动采纳
- 综合分 5.0-6.9 → 🟡 建议改造后采用
- 综合分 < 5.0 → 🔴 跳过

**使用方式**：

```bash
# 完整流程（扫描 + 分析 + 输出建议）
npm run evolve
# 或
/evolve run

# 只看候选
npm run evolve:candidates
# 或
/evolve candidates

# 实现第 1 个候选
npm run evolve:implement 1

# 检查已实现的特性是否过时
npm run trend
# 或
/evolve watch

# 生成趋势报告
npm run evolve:report

# 自我分析：看关键词/权重优化建议
npm run evolve:analyze-self
```

**实测效果**（首次扫描）：

- 🟢 采纳：7 个
- 🟡 改造：11 个

- Top 3：
  1. MemTensor/MemOS（7.95/10）— Self-evolving memory OS
  2. mksglu/context-mode（7.85/10）— Context window optimization
  3. ruvnet/ruflo（7.65/10）— Agent meta-harness

**防闭门造车机制**：
- 每日轻量检查：对比已实现特性 vs GitHub 新方案
- 每周深度对比：检测是否有更强替代方案
- 月度战略审计：分析社区趋势，输出报告

**相关文档**：

### 3.4 Agents 子代理使用

子代理用于复杂任务的**隔离执行**——它们在独立的 context 中运行，不会污染主对话。

| 代理 | 用途 | 适用场景 | 模型 |
|:-----|:-----|:---------|:-----|
| **planner** | 架构规划与技术选型 | 系统设计、方案评审 | Opus |
| **reviewer** | 多维度代码审查 | Bug 发现、安全检查 | Opus |
| **explorer** | 代码库探索研究 | 理解项目结构、查找代码 | Opus |
| **qa-reviewer** | 质量验证 | 回归测试、集成验证 | Opus |

使用方式：

```
用 planner 帮我设计这个系统的架构
用 reviewer 审查最近的变更
用 explorer 研究这个目录的代码结构
```

最佳实践：
- ✅ 复杂任务用子代理 → 保持主 context 干净
- ✅ reviewer 审查 planner 的方案 → 多角度验证
- ❌ 简单任务（如修一个变量名）不需要子代理
