## 🧬 零、智能演进路径 L1→L5（用户视角）

> **为什么这个工程叫"演进"而不是"工具集"**？因为所有功能不是平铺的，而是按"AI 越来越智能"的递进顺序组织的——**L1 工具能用 → L2 记得住 → L3 会决策 → L4 会学习 → L5 自治运行**。每一级都建立在前一级之上。

### 🗺️ L1→L5 全景图

```
L5 自治运行 ← 让 Claude 自主完成选题→实施→验证→复盘 全链路（v3.0.0 P0-1）
 ↑
L4 会学习 ← 让 Claude 从 GitHub 自主学习新能力（v2.2.0 M7 + LLM-judge M12 + 蒸馏 M13 + 指标 M15 + 图谱反哺 M14）
 ↑
L3 会决策 ← 让 Claude 自己判断"派不派 Agent"（v2.5.0 M9 + M10）
 ↑
L2 记得住 ← 让 Claude 跨会话记住一切（v2.1.0 M6 向量语义检索 + 左脑 KB）
 ↑
L1 工具能用 ← 让 Claude 能调 MCP / 文件 / 数据库（v1.x 基础设施）
```

## 🧠 二、核心能力详解

### 0. 核心定位：客户端 Agent 增强，不是大模型增强

**我们不做的事**：
- 训练/微调 Claude 模型
- 修改 Anthropic 的模型权重
- 替代 Claude 的推理能力

**我们做的事**：
- 在 Claude Code 客户端外面包了一层"增强壳"
- 让 Claude Code 这个 Agent 更会"使唤" Claude 模型

```
┌─────────────────────────────────────────┐
│           我们的增强层（AiCode）           │
│  调度器 / 记忆库 / 工具链 / 工作流 / 自动化   │
├─────────────────────────────────────────┤
│           Claude Code（官方客户端）         │
│  Agent 运行时 / Hook 系统 / 工具调用        │
├─────────────────────────────────────────┤
│           Claude 大模型（Anthropic）        │
│  推理 / 编码 / 理解 / 生成                 │
└─────────────────────────────────────────┘
```

#### 0.1 本质区别

| 维度 | 大模型增强 | 客户端 Agent 增强（我们的工作） |
|:-----|:-----------|:------------------------------|
| **改什么** | 模型权重、训练数据、微调、Prompt 工程、RAG | 调度、记忆、工具、工作流、UI、自动化 |
| **在哪里跑** | 云端/GPU | 本地 Claude Code CLI |
| **要不要算力** | 需要大量算力 | 几乎不要 |
| **谁能做** | 需要模型训练能力 | 普通开发者用脚本就能做 |
| **我们的项目** | ❌ 不做 | ✅ 全做 |

#### 0.2 每个系统对应 Agent 的哪个部分

| 我们的系统 | 属于 Agent 的哪个组件 | 作用 |
|:-----------|:----------------------|:-----|
| `dispatcher.js` 智能调度 | **编排层（Orchestration）** | 决定什么任务派子 Agent，什么自己干 |
| `left-brain` 记忆系统 | **长期记忆（Long-term Memory）** | 跨会话保存知识、偏好、决策 |
| `mcp.json` + MCP servers | **工具链（Tools）** | 给 Claude 提供本地文件/SQLite/网页抓取能力 |
| `hooks/` + `settings.local.json` | **环境集成（Environment Integration）** | 会话启动/结束/工具调用前自动执行逻辑 |
| `自我约束规范.md` | **工作流（Workflow）** | 标准化 AI 改完代码后的收尾流程 |
| `自我进化循环系统` | **学习反馈（Learning/Feedback）** | 让 Agent 从 GitHub 社区学习新能力 |
| `.claudeignore` + `/compact` | **上下文管理（Context Management）** | 控制传给模型的上下文，省 token |
| `qa-reviewer` + 测试 | **验证层（Validation/Safety）** | 独立验证代码质量，防止乱改 |
| `/evolve`, `/dispatch` 命令 | **用户体验层（UX）** | 给用户更高效的交互入口 |
| `会话快照` + `00_ROOT_快速加载会话.md` | **状态恢复（State Recovery）** | 让 Agent 能跨会话继续工作 |

#### 0.3 为什么这个定位更务实

**大模型我们做不了**：
- 训练/fine-tune 需要算力和数据
- 我们不可能比 Anthropic 更懂 Claude
- 模型能力由官方决定，我们只能等更新

**客户端 Agent 有巨大优化空间**：

| 官方能力 | 缺什么 | 我们补什么 |
|:---------|:-------|:-----------|
| 能对话 | 不会自己决定要不要派 Agent | 智能调度 |
| 有上下文窗口 | 跨会话失忆 | 左脑记忆 |
| 能调工具 | 没有本地记忆/数据库工具 | MCP sqlite/fetch |
| 能改文件 | 改完不自动测试/归档 | 自我约束 |
| 能联网 | 不会主动学习社区方案 | 自我进化 |
| 能并行 | 多个任务容易冲突 | worktree 隔离 |

#### 0.4 客户端增强是乘数效应

同样的 Claude 模型：
- 裸用 Claude Code = 1x
- + 智能调度 = 2-3x
- + 记忆 + 工具 + 工作流 = 5x
- + 自我进化 = 持续增长

#### 0.5 对比主流 Agent + LLM

| 维度 | 主流 Agent + LLM | 我们的 AiCode |
|:-----|:-----------------|:--------------|
| 学习能力 | 固定训练数据 | ✅ 每日扫描 GitHub 自我进化 |
| 调度策略 | 无/简单 | ✅ 规则引擎智能决策 |
| 跨会话记忆 | 对话历史 | ✅ 左脑知识库 + 图谱 |
| 收尾质量 | 需人工提醒 | ✅ 自我约束自动收尾 |
| 工程闭环 | 直接改文件 | ✅ 计划→迭代→归档 |
| 并行隔离 | 共享文件系统 | ✅ worktree 真隔离 |

### 2.24 任务复杂度评分驱动 Agent 数量（M9 + M10 v2.5.1）⭐⭐

**核心问题**：dispatcher 派/不派决策靠规则匹配 + 启发式，**没有量化**；M9 之前 `decide()` 多个路径仍固定派 2 个 Agent，**没有把评分转化为调度粒度**。

**解决方案**（M9）：`scoreComplexity(task) → { score, band, breakdown }` 0-10 数字评分。

**三档阈值**（`RULES.scoring`）：
- `score ≤ 3` → `no_dispatch`（不值得派）
- `4 ≤ score ≤ 7` → `gray_zone`（灰区，保守派）
- `score ≥ 8` → `dispatch`（直接派）

**加性打分**：
- 文件数 × 0.6 + 模块数 × 1.2（基础分）
- `should_dispatch` 关键词 +3（强信号）
- `dont_dispatch` 关键词 -1.5（抑制）
- 强任务类型（bug_fix / refactor / migration 等）+2
- 单人处理类型（explanation / question / single_edit）-1
- 钳制到 0-10

**M10 接入 Agent 数量**：`agentsFromScore(score) = Math.min(max_agents, Math.ceil(score / 3))`
- score 1-3 → 1 agent
- score 4-6 → 2 agents
- score 7-10 → 3 agents

**3 处固定 `agents: 2` 改为动态**：
- 命中 `should_dispatch` 关键词路径
- 任务类型匹配路径（bug_fix / refactor / migration 等）
- 灰区 `suggested_action.agents`

**关联**：
- 增量 B（plan-bridge）评分作为 plan 风险评估输入
- 自主模式评分作为"是否进入灰区自动选"依据
- dispatcher 决策日志供 metrics.js 统计"派/不派比"

### 2.28 Tool output sandbox（M26 · v3.0.5）⭐⭐⭐

> **目标**：借鉴 mksglu/context-mode 的 sandbox 思路，大工具输出压缩为"首 30 + 末 30 + 元信息"，单次减少 60-96% 字符。
> **痛点**：Claude Code Read/Bash/Grep 大输出塞满上下文窗口，token 消耗高、注意力分散。

**借鉴评估**（来自 research-skill-ecosystem-20260626 + candidates.json）：
- ⭐ 7.8/10（实用性 10 · 可行性 7 · 风险 2）
- 借鉴范围：**仅 sandbox 这块**；17 平台路由/MCP server 不动（避免范围蔓延）

**核心能力**：

| 函数 | 作用 |
|:-----|:-----|
| `shouldSandbox(text)` | > 5000 字符触发（可自定义阈值）|
| `sandboxOutput(text, opts)` | 保留首 30 行 + 末 30 行 + 中间省略标记 + 元信息 |
| `analyzeReduction(text)` | 触发判断 + 减少率统计（CLI `--analyze`）|

**真实效果**（demo）：
- 04.md（1780 行 / 64K 字符） → 60 行 + 元信息
- **减少 96%**（61,806 字符 → 2,374 字符）
- 保留关键信息：开头上下文 + 结尾结果 + 元信息（保留/省略/原始/节省）

**安全约束**（POC 阶段）：
- ✅ 纯函数无副作用
- ✅ CLI 演示模式（`node sandbox-tool-output.js <file>` / `--analyze`）
- ❌ **不接 PostToolUse hook**（避免误伤关键输出）
- ❌ **不替换 Read/Bash/Grep 默认行为**

**后续路径**（数据驱动）：
1. **POC 试用 1 周**：用 `--analyze` 模式观察哪些输出真被压缩
2. **决定是否接 hook**：如果没问题 → M27 接 PostToolUse 自动 sandbox
3. **渐进式**：先只对 `Bash` 输出（最常见），Read/Grep 留给后续增量

**L5 终极智能影响**：
- 直接降低 token 消耗 → L5 第 4 条（月度 metric）的 `task-time` 指标受益
- 为"AI 越用越省"提供基础设施（跟"AI 越用越聪明"互补）

**关联**：
- 引擎：`scripts/orchestrator/sandbox-tool-output.js`（120 行）

### 3.4 Agents 子代理使用

子代理用于复杂任务的**隔离执行**——它们在独立的 context 中运行，不会污染主对话。

| 代理 | 用途 | 适用场景 | 模型 |
|:-----|:-----|:---------|:-----|
| **planner** | 架构规划与技术选型 | 系统设计、方案评审 | Opus |
| **reviewer** | 多维度代码审查 | Bug 发现、安全检查 | Opus |
| **explorer** | 代码库探索研究 | 理解项目结构、查找代码 | Opus |
| **qa-reviewer** | 质量验证 | 回归测试、集成验证 | Opus |

使用方式：

```
用 planner 帮我设计这个系统的架构
用 reviewer 审查最近的变更
用 explorer 研究这个目录的代码结构
```

最佳实践：
- ✅ 复杂任务用子代理 → 保持主 context 干净
- ✅ reviewer 审查 planner 的方案 → 多角度验证
- ❌ 简单任务（如修一个变量名）不需要子代理

### 3.5 Hooks 自动化机制

系统配置了 4 个 hook，在后台自动运行：

| Hook | 触发时机 | 作用 |
|:-----|:---------|:-----|
| **PreToolUse** | 每次工具调用前 | 异步记录 Skill/Bash/Edit/Write 到日志 |
| **Setup** | 会话启动时 | 加载 MEMORY.md 索引 + 最近 5 条知识 |
| **PostToolUse** | Edit/Write 后 | 预留自动格式化入口 |
| **Stop** | Claude 停止时 | 检查感知队列 + 自动保存会话快照 |

感知队列：当不方便立即执行 `remember` 时：

```bash
bash .claude/skills/left-brain/scripts/enqueue.sh "要记忆的内容"
```

Stop hook 会在停止时提醒你处理。
