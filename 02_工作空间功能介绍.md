## 🧬 零、智能演进路径 L1→L5（用户视角）

> **为什么这个工程叫"演进"而不是"工具集"**？因为所有功能不是平铺的，而是按"AI 越来越智能"的递进顺序组织的——**L1 工具能用 → L2 记得住 → L3 会决策 → L4 会学习 → L5 自治运行**。每一级都建立在前一级之上。

### 🗺️ L1→L5 全景图

```
L5 自治运行 ← 让 Claude 自主完成选题→实施→验证→复盘 全链路（v3.0.0 P0-1）
 ↑
L4 会学习 ← 让 Claude 从 GitHub 自主学习新能力（v2.2.0 M7 + LLM-judge M12 + 蒸馏 M13 + 指标 M15 + 图谱反哺 M14）
 ↑
L3 会决策 ← 让 Claude 自己判断"派不派 Agent"（v2.5.0 M9 + M10）
 ↑
L2 记得住 ← 让 Claude 跨会话记住一切（v2.1.0 M6 向量语义检索 + 左脑 KB）
 ↑
L1 工具能用 ← 让 Claude 能调 MCP / 文件 / 数据库（v1.x 基础设施）
```

## 🧠 二、核心能力详解

### 0. 核心定位：客户端 Agent 增强，不是大模型增强

**我们不做的事**：
- 训练/微调 Claude 模型
- 修改 Anthropic 的模型权重
- 替代 Claude 的推理能力

**我们做的事**：
- 在 Claude Code 客户端外面包了一层"增强壳"
- 让 Claude Code 这个 Agent 更会"使唤" Claude 模型

```
┌─────────────────────────────────────────┐
│           我们的增强层（AiCode）           │
│  调度器 / 记忆库 / 工具链 / 工作流 / 自动化   │
├─────────────────────────────────────────┤
│           Claude Code（官方客户端）         │
│  Agent 运行时 / Hook 系统 / 工具调用        │
├─────────────────────────────────────────┤
│           Claude 大模型（Anthropic）        │
│  推理 / 编码 / 理解 / 生成                 │
└─────────────────────────────────────────┘
```

#### 0.1 本质区别

| 维度 | 大模型增强 | 客户端 Agent 增强（我们的工作） |
|:-----|:-----------|:------------------------------|
| **改什么** | 模型权重、训练数据、微调、Prompt 工程、RAG | 调度、记忆、工具、工作流、UI、自动化 |
| **在哪里跑** | 云端/GPU | 本地 Claude Code CLI |
| **要不要算力** | 需要大量算力 | 几乎不要 |
| **谁能做** | 需要模型训练能力 | 普通开发者用脚本就能做 |
| **我们的项目** | ❌ 不做 | ✅ 全做 |

#### 0.2 每个系统对应 Agent 的哪个部分

| 我们的系统 | 属于 Agent 的哪个组件 | 作用 |
|:-----------|:----------------------|:-----|
| `dispatcher.js` 智能调度 | **编排层（Orchestration）** | 决定什么任务派子 Agent，什么自己干 |
| `left-brain` 记忆系统 | **长期记忆（Long-term Memory）** | 跨会话保存知识、偏好、决策 |
| `mcp.json` + MCP servers | **工具链（Tools）** | 给 Claude 提供本地文件/SQLite/网页抓取能力 |
| `hooks/` + `settings.local.json` | **环境集成（Environment Integration）** | 会话启动/结束/工具调用前自动执行逻辑 |
| `自我约束规范.md` | **工作流（Workflow）** | 标准化 AI 改完代码后的收尾流程 |
| `自我进化循环系统` | **学习反馈（Learning/Feedback）** | 让 Agent 从 GitHub 社区学习新能力 |
| `.claudeignore` + `/compact` | **上下文管理（Context Management）** | 控制传给模型的上下文，省 token |
| `qa-reviewer` + 测试 | **验证层（Validation/Safety）** | 独立验证代码质量，防止乱改 |
| `/evolve`, `/dispatch` 命令 | **用户体验层（UX）** | 给用户更高效的交互入口 |
| `会话快照` + `00_ROOT_快速加载会话.md` | **状态恢复（State Recovery）** | 让 Agent 能跨会话继续工作 |

#### 0.3 为什么这个定位更务实

**大模型我们做不了**：
- 训练/fine-tune 需要算力和数据
- 我们不可能比 Anthropic 更懂 Claude
- 模型能力由官方决定，我们只能等更新

**客户端 Agent 有巨大优化空间**：

| 官方能力 | 缺什么 | 我们补什么 |
|:---------|:-------|:-----------|
| 能对话 | 不会自己决定要不要派 Agent | 智能调度 |
| 有上下文窗口 | 跨会话失忆 | 左脑记忆 |
| 能调工具 | 没有本地记忆/数据库工具 | MCP sqlite/fetch |
| 能改文件 | 改完不自动测试/归档 | 自我约束 |
| 能联网 | 不会主动学习社区方案 | 自我进化 |
| 能并行 | 多个任务容易冲突 | worktree 隔离 |

#### 0.4 客户端增强是乘数效应

同样的 Claude 模型：
- 裸用 Claude Code = 1x
- + 智能调度 = 2-3x
- + 记忆 + 工具 + 工作流 = 5x
- + 自我进化 = 持续增长

#### 0.6 用打游戏比喻

- **大模型** = 游戏角色的基础属性（力量/智力）
- **客户端 Agent** = 玩家的操作界面、快捷键、宏、背包管理、自动寻路、任务追踪

### 2.2 快照系统（v1.1，v1.9 升级：模式可控）

**核心问题**：会话太长 context 爆炸，下次开会话忘记上次的进度；快照保存"频繁但不可控"。

**解决方案**：一键备份 + 一秒恢复 + **4 模式会话级切换**

```
# 结束会话前（一句话）
node scripts/会话快照/save.js "任务标题" "中文标签" -m "下一步"

# 复杂任务：开干前先关，写完一个功能再开
/snap-mode off                    # 关闭自动保存
/snap-mode manual                 # 只手动 save.js
/snap-mode milestone              # 命中完成/交付才存（默认）
/snap-mode auto                   # 每次 Stop hook 都存
/snap-mode reset                  # 回到全局配置

# 下次开会话（一键）
node scripts/会话快照/load.js v1.1    # 按版本匹配
cat 00_ROOT_快速加载会话.md                # 看根目录索引
```

**4 模式详解**：

| 模式 | Stop hook | 手动 save.js | 显式调用 + --force |
|:-----|:----------|:-------------|:--------------------|
| `off` | 不存 | 不存 | **存** |
| `manual` | 不存 | 存 | 存 |
| `milestone` | 仅含"完成/里程碑/交付"等关键词时存 | 仅含关键词时存 | 存 |
| `auto` | 每次都存 | 存 | 存 |

**优先级链**：`.claude/session-snap-mode.json`（会话级） > `.claude/snapshot-config.json`（全局） > 默认 milestone

**自动化**：
- save.js 自动追加 00_ROOT_快速加载会话.md 的"快照列表"表格
- 自动加 ⭐ 最新 标记
- 自动加"快速启动命令"代码块
- 检测"完成/里程碑/交付"标签 → 加三级检查点提示
- **v1.9 修复**：milestone 关键词检查对显式调用也生效（之前显式调用绕过，是个 bug）

**状态栏实时可见**：

# 或

# 查看状态

### 2.20.1 高频使用场景（4 类用户路径 · v3.0.7）

| 场景 | 用户动作 | 关键认知 |
|:-----|:---------|:---------|
| **1 · 当前窗口做** | `/autonomous` → 选 always → 弹"启动方式"菜单 → 选"当前会话做" | Claude 在当前窗口干 4-5h，**窗口不能关** |
| **2 · 后台无人值守** | `/autonomous` always → 另开 PowerShell 跑 `npm run autonomous:runner` | runner 后台循环，**关窗口都行** |
| **3 · 新窗口接续** | 新窗口 → `/autonomous` always | SessionStart 自动加载快照 + 接 next 队首，**不会自动启动 runner**（仍需手动 `npm run autonomous:runner`） |
| **4 · 中途切 runner** | 当前 Claude 干 30 分钟 → Esc 取消 → 切 PowerShell 跑 `npm run autonomous:runner` | 切换瞬间无工作丢失（快照 + 锁机制 + evolution-plan.json current 字段） |

**5 条关键认知（避坑指南）**：
1. `/autonomous` **只是开关**，不是启动跑——runner 永远要手动 `npm run autonomous:runner`
2. `always` ≠ "在跑"——`always` 是"允许循环"，runner 是"实际循环"
3. **新窗口 + /autonomous** 不会自动干活——只开开关
4. **always 自动接力** 下一个候选，runner 跑完一个阶段后
5. **always 空转保护**：队列空了 always 自动 OFF（避免无限空转）

**和 `/autonomous-stop` 关系**：
- `single` 模式 = 跑完 1 阶段**自动** OFF
- `always` 模式 = 一直跑直到你**手动** OFF 或队列空
- 任何时候想停 → `/autonomous-stop` 或 `npm run autonomous:off`

**关联**：
- `01.md` §三"🚀 自主模式高频场景"（用户速查版）
- `.claude/commands/autonomous.md`（命令字典，**待建 M51**）
- `04.md` §0.4 M48 增量段（升级动机）

**安全边界**：
- ✅ 自主做：智能增量深化、bug 修、文档、commit
- ⚠️ 慎做：修改 `scripts/orchestrator/`、`.claude/`、CLAUDE.md（commit 前先 snapshot）
- ❌ 不做：push、删分支、删文件、改主目录外文件

### 2.30 Multi-Agent Swarm 协调（借鉴 ruvnet/ruflo · M31 · v3.0.5）⭐⭐⭐

> **目标**：借鉴 ruvnet/ruflo 的 "agent meta-harness + multi-agent swarm"，任务派出去后**汇总 + 投票**，让复杂任务多视角分析。
> **痛点**：AiCode dispatcher 只决定"派不派 + 派几个"，派出去的 Agent 各自独立回答，没有"汇总 + 投票"机制。复杂任务单 Agent 视角容易盲。

**借鉴评估**：
- ⭐ 7.7/10（实用性 10 · 可行性 7 · 风险 2）
- 借鉴范围：**多视角生成 + 投票汇总**；adaptive memory / RAG / 自学习 swarm 不动

**核心能力**（3 个纯函数）：

| 函数 | 作用 |
|:-----|:-----|
| `generatePerspectives(task, n)` | 生成 N 个异构视角 prompt（5 视角池：安全/性能/可维护性/简洁性/兼容性）|
| `aggregateResults(results, strategy)` | 3 种汇总策略（majority / weighted / best-of）|
| `swarmDecide(task, opts)` | 一站式入口（mock 演示 + runAgent 注入）|

**3 种策略对比**：

| 策略 | 算法 | 适用场景 |
|:-----|:-----|:---------|
| **majority**（默认）| Jaccard 相似度聚类 + 选最大一致组 | 多 Agent 输出措辞相似时 |
| **weighted** | 按每个结果的 score 求和取最大组 | Agent 自带置信度时 |
| **best-of** | 直接选 score 最高的 | 单 Agent 信任模式 |

**真实效果**（demo）：
- 任务：`重构 dispatcher.js 提升派 agent 准确率`（n=3）
- 输出：**3 个异构视角 prompt + 3 个模拟 Agent 输出 + 投票结果**

  - 视角 2 性能：算法复杂度、缓存策略、批量处理
  - 视角 3 可维护性：函数拆分、注释、单元测试覆盖
- 最终答案：`[安全性视角] 建议 ...`（majority 选第一个最大组）

### 3.4 Agents 子代理使用

子代理用于复杂任务的**隔离执行**——它们在独立的 context 中运行，不会污染主对话。

| 代理 | 用途 | 适用场景 | 模型 |
|:-----|:-----|:---------|:-----|
| **planner** | 架构规划与技术选型 | 系统设计、方案评审 | Opus |
| **reviewer** | 多维度代码审查 | Bug 发现、安全检查 | Opus |
| **explorer** | 代码库探索研究 | 理解项目结构、查找代码 | Opus |
| **qa-reviewer** | 质量验证 | 回归测试、集成验证 | Opus |

使用方式：

```
用 planner 帮我设计这个系统的架构
用 reviewer 审查最近的变更
用 explorer 研究这个目录的代码结构
```

最佳实践：
- ✅ 复杂任务用子代理 → 保持主 context 干净
- ✅ reviewer 审查 planner 的方案 → 多角度验证
- ❌ 简单任务（如修一个变量名）不需要子代理

### 3.5 Hooks 自动化机制

系统配置了 4 个 hook，在后台自动运行：

| Hook | 触发时机 | 作用 |
|:-----|:---------|:-----|
| **PreToolUse** | 每次工具调用前 | 异步记录 Skill/Bash/Edit/Write 到日志 |
| **Setup** | 会话启动时 | 加载 MEMORY.md 索引 + 最近 5 条知识 |
| **PostToolUse** | Edit/Write 后 | 预留自动格式化入口 |
| **Stop** | Claude 停止时 | 检查感知队列 + 自动保存会话快照 |

感知队列：当不方便立即执行 `remember` 时：

```bash
bash .claude/skills/left-brain/scripts/enqueue.sh "要记忆的内容"
```

Stop hook 会在停止时提醒你处理。
