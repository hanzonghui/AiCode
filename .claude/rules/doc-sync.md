# 📚 文档同步规则（Doc Sync）

> **作用**：里程碑或增量完成后，强制同步根目录下 4 个模板文档 + CHANGELOG，避免 04/03/CLAUDE/CHANGELOG 出现"已完成但文档还写计划中"的状态漂移。
> **触发节点**：每个里程碑（M1~Mn）/ 增量（A~G）/ 大版本发版（vX.Y.0）/ 路线图重写。
> **创建日期**：2026-06-25
> **背景**：04 文档 0.4 节 增量 E/F/G 已写"🆕 计划中"，但实际 M6/M7/M8 早已交付；里程碑表却已 ✅。**同类问题不能再发生**。

> 🚨 **2026-06-25 强化**：本规则被 `self-discipline.md` 决策树在 **🔴 大 / 🏁 里程碑** 级别**强制触发**（动作 4a），不再由 AI 自由判断。
> - 引用位置：`.claude/rules/self-discipline.md` 表格 + `scripts/orchestrator/自我约束规范.md` 决策树
> - 跳过条件：仅当明确判定"用户不可见"时（如纯内部重构），并在 commit 消息中说明

## 🎯 必须同步的 4 个根目录文档

| 文档 | 路径 | 同步内容 |
|:-----|:-----|:---------|
| **演进路线** | `04_自我演进路线.md` | 0.4 节增量状态（计划中→已完成/Mx 编号/commit）、0.5 长期愿景表 L1~L5、顶部"升级时间"、第十二章里程碑表 |
| **版本计划** | `03_版本迭代计划.md` | 第四节 P0/P1 完成状态、第七节发版流程记录、第十一节追加方法论（如涉及） |
| **工作空间说明** | `02_工作空间功能介绍.md` | 新增/修改的章节（2.X 节功能速览、现状速览表） |
| **CLAUDE.md** | `CLAUDE.md` | 工作空间结构树、核心能力表、规则文件清单、快速操作 |
| **CHANGELOG.md** | `CHANGELOG.md` | **每次必写**（放最顶部的 Unreleased 段） |

> 原则：**`CHANGELOG.md` 是源事实**（代码已完成→就写条目）；`04.md` / `03.md` 是上层状态汇总 → 反向同步 CHANGELOG；`CLAUDE.md` / `02.md` 是 UX 层 → 只在用户可见变化时同步。

## 🚦 触发节点（强制同步）

### 节点 1：完成一个增量（A/B/C/D/E/F/G/Mn）

```bash
# 提交后立即同步（4 文件并行）
# 1. CHANGELOG.md - 顶部 Unreleased 段
# 2. 04_自我演进路线.md - 0.4 节增量段 + 0.5 长期愿景表 + 里程碑表
# 3. 03_版本迭代计划.md - 对应 P0/P1 行状态
# 4. CLAUDE.md / 02 - 仅当用户可见变化时
```

### 节点 2：完成一个里程碑（Mx）

```bash
# 提交后立即同步
# 1. CHANGELOG.md - 版本号段
# 2. 04_自我演进路线.md - 第十二章里程碑表
# 3. 03_版本迭代计划.md - 整体进度概览
# 4. CLAUDE.md - 核心能力表（如新增能力）
```

### 节点 3：发版（vX.Y.0 → git tag）

```bash
# 完整重排
# 1. CHANGELOG.md - Unreleased → 正式版本号 + 日期
# 2. 04_自我演进路线.md - 顶部"升级时间"、0.4 节、0.5 节
# 3. 03_版本迭代计划.md - "当前版本"字段
# 4. CLAUDE.md - 全部版本号引用
# 5. package.json - version 字段
```

### 节点 4：路线图重写（如 v1.2 修订 → v2.0 改名）

```bash
# 全量重写 + 旧路线图归档到 archives/
# 5 文件全部同步，CLAUDE.md 顶部"启动协议"段也要重审
```

## 🤖 AI 自动检测（增量 C 集成）

> **集成位置**：`scripts/orchestrator/proactive/proactive-scan.js` 的 7 维度之一
> **维度名**：`doc-drift`（候选维度，v2.x 启用）
> **检测逻辑**：
>
> 1. 读 `CHANGELOG.md` 最近 1 个版本号 + 日期
> 2. 读 `04_自我演进路线.md` 顶部"最近一次同步"日期
> 3. 读 `04_自我演进路线.md` 第十二章里程碑表最后一行日期
> 4. 读 `03_版本迭代计划.md` 顶部"当前版本"字段

> 6. 进一步扫 04.md 0.4 节是否有 "🆕 计划中" 关键词（每出现一次 +1 漂移点）
> 7. 输出 anomaly 报告："文档漂移：M6/M7/M8 已交付但 04.md 还写'计划中'"

**触发频率**：SessionStart 时跑（参考增量 C 的 5 分钟缓存）

## ✅ 自我约束（Self-Discipline 联动）

> 文档同步是 **🏁 里程碑** 级别动作的必选项，对应 `.claude/rules/self-discipline.md` 的"文档更新"。

| 级别 | 同步要求 |
|:-----|:---------|
| 🟢 微小 | ❌ 跳过 |
| 🟡 小 | ❌ 跳过（除非新增规则文件） |
| 🔴 大 | ✅ 必做：CHANGELOG + 04 + 03（CLAUDE/02 按需） |
| 🏁 里程碑 | ✅ 必做：5 文件全同步（CHANGELOG + 04 + 03 + CLAUDE + 02） |

**违反该规则**：
- AI 完成的代码与文档不一致 → 增量 A 自我反思会捕获（"CHANGELOG 与 04.md 状态冲突"）
- 主动扫描会提示（"文档漂移 anomaly"）
- 用户审查时会发现 → 信任损耗
