# Node.js spawn 在 Windows 找不到全局 npm CLI 的根因

## 问题
2026-06-25 启动 `/autonomous single` → runner 报 `spawn claude ENOENT` × 5 次。但 `where.exe claude` 在 PowerShell 里能找到 `C:\Users\Administrator\AppData\Roaming\npm\claude.cmd`，CLI 实际已安装。

## 根因
Node.js `child_process.spawn(cmd, args, { shell: false })` 在 Windows 上：
- 不继承父 PowerShell 进程的 PATH
- 直接调用 Win32 API `CreateProcess`，只搜 `cmd.exe` 的标准 PATH 查找逻辑
- 全局 npm 目录 `%APPDATA%\npm\` 不在系统 PATH 中（PowerShell 通过用户配置文件注入），所以 ENOENT

## 修复方案（按推荐度）
1. **`shell: true`**（最简单）：让 PowerShell/cmd 接管 PATH 解析，但要把 args 数组转字符串（数组 + shell:true 在 Windows 上工作良好）
2. **绝对路径 fallback**：用 `path.join(process.env.APPDATA, 'npm', 'claude.cmd')` + `fs.existsSync()` 显式定位
3. **CLAUDE_BIN 环境变量**：给高级用户覆盖入口

生产代码建议 **1 + 2 组合**（spawn shell:true + 默认值优先用绝对路径）。

## 教训
- Windows 上 Node spawn 是常见坑，写跨平台工具必须实测
- ENOENT 不一定是真的"没装"，可能是"找不到"
- 给清晰 hint（"请 npm i -g @xxx" / "设置 ENV 变量"）比沉默失败友好 10 倍

## 验证
v2.2.1 修复后 `resolveClaudeBin()` 返回 `C:\Users\Administrator\AppData\Roaming\npm\claude.cmd`，exists=true。测试 8/8 通过。
