## 二、核心系统设计

> 详细设计见：[04_自我进化循环系统设计.md](./04_自我进化循环系统设计.md)

### 2.1 自我进化循环

```
GitHub Trending + Search API
  ↓
trending.json（每日快照，保留7天）
  ↓
评估引擎（规则 + LLM 打分）
  ↓
candidates.json（评分 + 理由 + 复杂度）
  ↓
实现引擎（创建分支 → 写代码 → 测试 → 审查）
  ↓
QA 验证 + 合并 + 左脑记忆 + evolved-features.json
  ↓
持续感知层（每日/每周/月度对比，防退化）
  ↓
发现更好方案 → 升级建议 → 回到实现流程
```

### 2.2 评估维度

| 维度 | 权重 | 标准 |
|:-----|:-----|:-----|
| 实用性 | 30% | 对工作流有什么帮助 |
| 可行性 | 25% | 现有架构能否实现 |
| 独立性 | 20% | 是否依赖外部服务 |
| 风险度 | 15% | 引入后是否破坏现有系统 |
| 新鲜度 | 10% | 是否是我们还没有的能力 |

### 2.3 决策规则

- `composite >= 7.0` → 🟢 自动采纳（P0）
- `composite 5.0-6.9` → 🟡 建议采纳（需确认）
- `composite < 5.0` → 🔴 跳过

### 2.4 安全约束（v1.2 修订）

> ⚠️ **修订原因**：原约束"⛔ 不修改 dispatcher.js 核心"会和本文件 v1.9 P0-2 加 withRetry 冲突。
> 自我进化系统不能修改 dispatcher 核心，但**人工迭代可以**——这是两类不同活动。

| 约束 | 适用范围 |
|:-----|:---------|
| ⛔ 不引入 npm 依赖（除非绝对必要） | 全部 |
| ⛔ 不修改 dispatcher.js 核心 | **仅自我进化系统** |
| ⛔ 不删除现有文件 | 全部 |
| ✅ 必须通过 `npm test` | 全部 |
| ✅ 新功能必须带测试 | 全部 |
| ✅ 新功能必须更新文档 | 全部 |

**新增约束**（v1.9 起）：
- ✅ P0 基础设施项必须有 GitHub Actions 验证
- ✅ 涉及跨模块改动必须有 metrics 增量
- ✅ 任何权限/审计相关改动必须在 audit log 落盘

## 十一、附录：升级方法论（v1.9.0 新增）

> **来源**：原 `06_差距分析与升级路径.md`（2026-06-24 创建，2026-06-24 合并到本文件）
> **目的**：沉淀"从 L2 到 L4 该怎么走"的方法论，避免每次发版重新发明

### 11.1 级别定义（统一标尺）

| 级别 | 名称 | 关键特征 | 典型例子 |
|:-----|:-----|:---------|:---------|
| L1 | 玩具 | 几个脚本，无文档无测试 | 多数个人尝试 |
| L2 | 个人工程 | 有结构、有文档、有测试 | **AiCode v1.9 前** |
| L3 | 工程化个人项目 | L2 + CI + 自动测试 + 监控 | **AiCode v1.9 当前**、ripgrep / fd |
| L4 | 小团队产品 | L3 + 权限 + 审计 + 多环境 | 创业公司内部工具 |
| L5 | 商业产品 | L4 + SLA + 合规 + 商业化 | GitHub Copilot 客户端 |

> **L2 ↔ L3 的硬指标**：**"离开作者能不能跑起来"**（不是代码好不好）。

### 11.2 短板 ROI 排序法

每次发版前的 P0/P1/P2 排序遵循三个原则：

| 原则 | 含义 | 例子 |
|:-----|:-----|:-----|
| **生死线 > 锦上添花** | 不补就停摆的优先 | CI/CD > 新功能 |
| **基础设施 > 功能** | 地基不稳盖楼会塌 | metrics > MCP 工具扩展 |
| **L 级别跃迁 > 同级优化** | 跳级的杠杆最大 | L2→L3 跨 CI > L3 内优化代码 |

> v1.9 P0 排序示例：CI (L 跃迁) > 兜底 (L 跃迁) > MCP 错误 (L3 内) > Metrics (L 跃迁)

### 11.3 工作量估算模板

每个任务估时按下面 4 档：

| 档位 | 工作量 | 适合 |
|:-----|:-------|:-----|
| 半天 | 0.5 天 | 改配置、加文件、调 API |
| 一天 | 1 天 | 写测试、文档章节、单一工具 |
| 两天 | 2 天 | 新模块 + 测试 + 接入 |
| 一周 | 5 天 | 重构、新子系统、跨模块改造 |

> **超 1 周必拆**。v1.9 实际：13 项任务全在 0.5-2 天内，最长 2 天（withRetry + 接入）。

### 11.5 升级决策记录

#### 决策 4：v1.9 提前 4 周发版

- **时间**：2026-06-24
- **背景**：原计划 v1.9 用 4 周（4 周窗口）

#### 决策 5：MCP 工具扩展不进入 v1.9

- **选择**：v3.0 之前不做

### 11.6 v1.9.0 升级验证（已完成）
